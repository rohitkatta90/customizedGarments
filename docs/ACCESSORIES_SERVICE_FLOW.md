# Additional Styling Elements — service flow

## Category

**Additional Styling Elements** covers optional extras beyond **base stitching or alteration** — e.g. laces/borders, tassels/latkans, buttons/zippers, padding/cups, embroidery, dyeing/colour matching, extra lining/layers.

Presets live in `src/lib/orders/styling-elements.ts` (`STYLING_ELEMENT_PRESETS`). Staff can still type **custom** lines.

## Pricing logic

| Layer | Rule |
|-------|------|
| **Base** | Standard stitching/alteration as quoted in WhatsApp — **no** automatic inclusion of styling extras. |
| **Extras** | Identified from **design** and **discussion**; stored as **`quotedAccessories`** on the order. |
| **Charges** | Each line can have **₹ extra** (whole rupees) or **amount TBD** (null until confirmed). |
| **Proceed** | Work on extras only after **customer approval** — track with **`accessoriesQuoteStatus`**. |

Statuses: `none` → `pending_customer` → `approved` or `declined`.

### Indicative add-on bands (ranges, not fixed)

Public **`/pricing`** lists **per-category INR ranges** for laces, latkans, padding, embroidery, dyeing, extra lining, etc. (`public/data/pricing.json` → `stylingExtras`). Final **₹** still depends on quantity, complexity, and materials — see **[STYLING_EXTRAS_PRICING.md](./STYLING_EXTRAS_PRICING.md)**.

## Customer communication (site)

- **Gallery** and **Service request** show the same bilingual notice (`dict.styling.pricingNotice`): extras may incur charges and are **always confirmed before proceeding**.
- Public **tracking** does **not** list accessory quotes (keeps UI simple); confirmation stays on **WhatsApp**.

## Staff workflow (admin)

1. Open **Order details** (`/admin/orders/[id]`).
2. Under **Additional Styling Elements**, add lines (preset or free text), optional ₹ amounts.
3. Set **Extras approval status** (e.g. `pending_customer` after sending quote).
4. Click **Copy WhatsApp confirmation (extras)** — message lists lines, optional total, order ref, and *“Shall we proceed with these add-ons?”*
5. Paste in WhatsApp; after the customer agrees, set status to **`approved`** and save (and move main order status forward as usual).

Internal context goes in **Notes (internal)**; not shown to customers.

## Data model (Firestore)

| Field | Purpose |
|--------|---------|
| `quotedAccessories` | `[{ id, label, amountInr }]` |
| `accessoriesQuoteStatus` | `none` \| `pending_customer` \| `approved` \| `declined` |
| `accessoriesNotesInternal` | Staff-only |
